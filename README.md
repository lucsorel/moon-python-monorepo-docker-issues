# moon-python-monorepo-docker-demo

Sample moon monorepo to demonstrate some issues related to the building of Docker images for a Python application.
Python projects are handled by uv.

The demo application is a CLI tool taking a value between 0 and 1 and interpolating it in a given output interval.

Usage:

```sh
# with interpolate-cli: a Python standalone moon project
moon r cli:cli -- --value 0.5  --output-min 10.0 --output-max 20.0
# -> 15.0

# with interpolate-cli-dep: a Python moon project depending on a Python library project (interpolate-lib)
moon r cli-dep:cli -- --value 0.5  --output-min 10.0 --output-max 20.0
# -> 15.0
```

- [.moon/workspace.yml](.moon/workspace.yml)
- [.moon/toolchains.yml](.moon/toolchains.yml)
- root [moon.yml](moon.yml) commands to build the Docker images

## interpolate-cli: issues with a standalone python project

- moon project: [interpolate-cli/moon.yml](interpolate-cli/moon.yml)
- [Dockerfile.cli](Dockerfile.cli) and associated [Dockerfile.cli.dockerignore](Dockerfile.cli.dockerignore)

```sh
# build the Docker image
docker build -t "interpolate/cli:latest" -f Dockerfile.cli .

# runs the Docker image
docker run --rm interpolate/cli:latest -- --value 0.5  --output-min 10.0 --output-max 20.0
```

### Issue #1: `moon docker setup` requires the source code of the projects, but `moon docker scaffold` does not yield it

**Description**

The `build` stage in the moon-generated [Dockerfile.cli](Dockerfile.cli) file starts by copying the `.moon/docker/configs` contents and calls `moon docker setup`.

`.moon/docker/configs` contains only the `moon.yml` config file and some Python descriptors (`pyproject.toml` and `uv.lock`) of each moon project.

However, `moon docker setup` calls `uv sync --no-managed-python --no-python-downloads --no-progress`, which requires the `interpolate-cli/src/` source files and fails with this error:
```sh
> [build 2/4] RUN moon docker setup:

2.706 ▮▮▮▮ installing proto 0.56.4
4.711 ▮▮▮▮ installing unstable_python 3.12.13
9.818 ▮▮▮▮ installing unstable_uv latest

10.92 ▮▮▮▮ uv sync --no-managed-python --no-python-downloads --no-progress
11.07 Using CPython 3.12.13 interpreter at: /root/.proto/tools/python/3.12.13/bin/python3
11.07 Creating virtual environment at: .venv
11.07 Resolved 3 packages in 0.77ms
11.07    Building interpolate-cli @ file:///app/interpolate-cli
11.07   × Failed to build `interpolate-cli @ file:///app/interpolate-cli`
11.07   ╰─▶ Expected a Python module at: src/interpolate_cli/__init__.py
11.13 Error: process::failed
11.13
11.13   × Process uv failed: exit code 1
```

**Workaround**

Manually copying the source files in the `build` stage of the Dockerfile before running `moon docker setup`.

```Dockerfile
FROM base AS skeleton

COPY . .

RUN moon docker scaffold cli

FROM base AS build

COPY --from=skeleton /app/.moon/docker/configs .

# FIX moon docker setup builds interpolate-cli, and thus needs its source code
COPY --from=skeleton /app/interpolate-cli/src /app/interpolate-cli/src

RUN moon docker setup
```

**Expected behavior**

1. source files of a uv-managed project should be copied by `moon docker scaffold {python-project}` in `.moon/docker/configs`
2. **or** the Dockerfile generated by `moon docker file {python-project}` should contain the `COPY --from=skeleton /app/{python-project}/src /app/{python-project}/src` instruction

### Issue #2: `moon docker setup` requires the `README.md` files of the projects, but `moon docker scaffold` does not yield them

Very similar issue: the `uv sync --no-managed-python --no-python-downloads --no-progress` command called by `moon docker setup` requires the `README.md` file of the project if the `pyproject.toml` references it (see line 5 of [interpolate-cli/pyproject.toml](interpolate-cli/pyproject.toml)).

The workaround is also to manually copy it in the `build` stage:

```Dockerfile
FROM base AS skeleton

COPY . .
RUN moon docker scaffold cli

FROM base AS build

COPY --from=skeleton /app/.moon/docker/configs .
# FIX moon docker setup builds interpolate-cli, and thus needs its source code (and README.md file referenced in the pyproject.toml file)
COPY --from=skeleton /app/interpolate-cli/src /app/interpolate-cli/src
COPY --from=skeleton /app/interpolate-cli/README.md /app/interpolate-cli/README.md

RUN moon docker setup
```

### Issue #3: `moon r --no-actions --quiet` still implies actions and logs in a fully-built Docker image (because of `uv`)

The 2 previous workarounds permitted to build the Docker image.
Its ENTRYPOINT is the following moon task:

```Dockerfile
ENTRYPOINT [ "moon", "run", "--no-actions", "--quiet", "cli:cli" ]
```

```yaml
# interpolate-cli/moon.yml
tasks:
  cli:
    command: uv run interpolate-cli
    preset: server
```
**Description**

However running a container with the `--no-actions` and `--quiet` flags still seem to change the files structure and yield some logs:

```sh
docker run --rm interpolate/cli:latest -- --value 0.5  --output-min 10.0 --output-max 20.0

   Building interpolate-cli @ file:///app/interpolate-cli
      Built interpolate-cli @ file:///app/interpolate-cli
Uninstalled 1 package in 19ms
Installed 1 package in 4ms
15.0
```

Changing the ENTRYPOINT to add logs (`ENTRYPOINT [ "moon", "run", "--no-actions", "--quiet", "cli:cli" ]`) shows that the logs are likely to be generated by `uv`.
I just wonder why it needs to uninstall then install the project.

**Workaround**

A workaround in the `Dockerfile.cli` would be more complicated:
- set the `WORKDIR` to `/app/interpolate-cli`
- activate the virtual environment
- bypass `uv` by running a python command using the virtual environment
- which defeats the purpose of using a moon task and the python/uv toolchain

## interpolate-cli-dep: issues with a python project depending on another one

The `interpolate-cli-dep` project is very similar to `interpolate-cli`, it just imports an interpolation function defined in the `interpolate-lib` project of the monorepo.
`interpolate-lib` has numpy as a third-party production library.

- the [interpolate-cli-dep/moon.yml](interpolate-cli-dep/moon.yml) has a `dependsOn` clause on the `interpolate-lib` project.
- [Dockerfile.cli-dep](Dockerfile.cli-dep) and associated [Dockerfile.cli-dep.dockerignore](Dockerfile.cli-dep.dockerignore)

```sh
# build the Docker image
docker build -t "interpolate/cli-dep:latest" -f Dockerfile.cli-dep .

# runs the Docker image
docker run --rm interpolate/cli-dep:latest -- --value 0.5  --output-min 10.0 --output-max 20.0
```

### Issue #4 `moon docker setup` requires the source code and `README.md` files of dependent projects, but `moon docker scaffold` does not yield them

**Description**

Similarly to issues #1 and #2, we need to manually add the source code and the README.md files of `interpolate-cli-dep` and `interpolate-lib` for `moon docker setup` to execute successfully.

**Workaround**

See [Dockerfile.cli-dep](Dockerfile.cli-dep):

```Dockerfile
FROM base AS skeleton

COPY . .
RUN moon docker scaffold cli

FROM base AS build

COPY --from=skeleton /app/.moon/docker/configs .
# FIX moon docker setup builds interpolate-lib as a dependency of interpolate-cli, thus needs its source code
COPY --from=skeleton /app/interpolate-lib/src /app/interpolate-lib/src
COPY --from=skeleton /app/interpolate-lib/README.md /app/interpolate-lib/README.md
COPY --from=skeleton /app/interpolate-cli-dep/src /app/interpolate-cli-dep/src
COPY --from=skeleton /app/interpolate-cli-dep/README.md /app/interpolate-cli-dep/README.md

# Install toolchains and dependencies
RUN moon docker setup
```

### Issue #5 Docker images unnecessarily contain the virtual environments of dependent projects

**Description**

The `moon docker setup` builds the `interpolate-lib` project (by executing the `uv sync --no-managed-python --no-python-downloads --no-progress` command, which creates its virtual environment and downloads its dependencies), then builds the `interpolate-cli-dep` project (by creating its own virtual environment and downloading its own dependencies plus the ones of `interpolate-cli-dep`).

I believe `interpolate-lib` is built because this project is a moon dependency of the `interpolate-cli-dep` project.

The `interpolate-lib` virtual environment is irrelevant (useless) for the `interpolate-cli-dep` application, but is copied in the [Dockerfile.cli-dep](Dockerfile.cli-dep) `start` stage:

```Dockerfile
FROM base AS start

# ...

# /app/interpolate-lib/.venv is copied in the start stage
COPY --from=build /app /app
```

When using pandas with pyarrow (to handle dataframes, see [here](https://pandas.pydata.org/docs/user_guide/pyarrow.html)), it adds 150Mb to the Docker image.

To inspect the container from inside:

- comment out `RUN rm -fr /app/interpolate-lib/.venv` in [Dockerfile.cli-dep](Dockerfile.cli-dep)
- build the Docker image `docker build --no-cache -t "interpolate/cli-dep:latest" -f Dockerfile.cli-dep .` or `moon r docker-build-dep`
- run interactively the Docker image:

```sh
docker run --rm -it --entrypoint /bin/bash interpolate/cli-dep:latest

cd /app/interpolate-lib/.venv
ls -la
```

**Workaround**

Manually remove the `interpolate-lib` virtual environment at the end of the `build` stage:

```Dockerfile
FROM base AS build

COPY --from=skeleton /app/.moon/docker/configs .
# ...

RUN moon docker setup
COPY --from=skeleton /app/.moon/docker/sources .
RUN moon docker prune

# (FIX) deletes a useless virtual environment
RUN rm -fr /app/interpolate-lib/.venv

#### START STAGE
FROM base AS start

COPY --from=build /app /app
```
